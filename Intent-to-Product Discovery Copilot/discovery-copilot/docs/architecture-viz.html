✦ Architecture Blueprint

Discovery Copilot Architecture

Complete product architecture for an AI-powered decision engine that converts vague shopping intent into ranked, explainable product recommendations.

Executive Summary

Discovery Copilot is a decision-oriented AI search product for e-commerce. Not a chatbot. Not semantic search. A system that takes a vague shopping query and returns: a best pick, ranked alternatives, evidence-grounded explanations, confidence scoring, and tradeoff disclosure.

Product Definition

Mission: Make "what should I buy?" answerable by AI — with evidence, confidence, and honesty.

Core promise: The user enters a shopping need in natural language. The system returns a decision-quality recommendation, not just a list of results.

Decision output format

{
  "decision": {
    "bestPick": { ... },
    "alternatives": [ ... ],
    "rationale": "Prioritized comfort + low return risk for standing use case",
    "tradeoffSummary": "Best Pick is $20 more but has 2x better return rate",
    "missingInformation": ["shoe size", "budget"]
  },
  "confidence": { "overall": 0.87, "recommendation": "high_confidence" },
  "suggestedRefinements": ["Under $100", "Prefer leather"]
}

Business models

API / Embedded mode

Merchants integrate via REST API or JS SDK. Full control over UI. Single POST /api/v1/decide endpoint returns the complete decision.

Best for: custom storefronts, mobile apps, headless commerce

Hosted / SaaS mode

Managed search page with merchant branding. Drop-in widget. Analytics dashboard. Deploy in a day with just a product feed.

Best for: Shopify stores, SMBs, rapid deployment

Core User Experience

The UX is structured as: Search input → Decision display. Not a chat. The primary interface is a search bar that returns a decision card — visually dominant best pick, with alternatives below.

Flow by user type

UserFlowClarification
First-time visitorQuery → Decision with explanations0–1 questions max
Repeat logged-in userMemory-enriched → personalized rankingFewer questions (memory fills gaps)
High ambiguity query1–2 inline questions → then decisione.g., "a couch" → "What size room?"
Gift shopperRecipient profiling + budget → decision"What's he into?" + budget question
Fit-sensitive queryDecision + prominent size/fit warnings"What size?" if not specified

What appears above the fold

1. Query acknowledgment ("Best products for comfortable shoes for standing all day")
2. Confidence indicator (High / Moderate / Low)
3. Best Pick card (large, visually dominant, with explanation)
4. Top 2–3 alternatives visible without scrolling
5. Decision rationale sidebar (desktop) or inline (mobile)

Clarification strategy

6-factor weighted decision framework. Score < 0.3 → show results. 0.3–0.5 → show + refine. 0.5–0.7 → ask 1–2 questions. > 0.7 → ask 2–3 questions first.

FactorWeight
Ambiguity (1 − confidence)0.25
Attribute sensitivity (size/fit/skin)0.20
Consequence of mismatch0.20
Catalog breadth0.15
Budget uncertainty0.10
Confidence spread0.10

Override: After 2 turns without showing products → ALWAYS show results.

Final Recommendation

Start with API mode + one design-conscious merchant

Fastest path to demo: 48-hour prototype with hardcoded catalog, 5 example queries, premium UI.
Best business model to start: API-first (proves the decision engine), add hosted mode in Phase 2.
Build first: Decision API (/decide endpoint) + premium Best Pick card + 10 example queries.
Postpone: Memory, multi-tenant admin, hosted dashboard, review intelligence.
Biggest tech risk: Latency — multi-stage pipeline must stay under 3 seconds.
Biggest product risk: Decision quality — if the "best pick" is wrong, trust evaporates instantly.
Biggest UX risk: Clarification fatigue — ask too many questions and users leave.
